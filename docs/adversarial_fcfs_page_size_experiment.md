# Adversarial FCFS Page-Size Experiment

## Goal

The earlier `fcfs_static_prefix` experiments made delayed reuse too easy for `LRU` by clustering same-family requests closely together. This experiment instead uses delayed-reuse request orders that are still realistic, but less friendly to pure recency.

The question is:

`Under deterministic FCFS scheduling, do delayed-reuse workloads and coarser page sizes create a materially larger Belady-vs-LRU gap?`

## Workloads

Generated by:

- [generate_adversarial_fcfs_workloads.py](/Users/tejguntuku/TEJ/CS_Independent_Research/kv_cache_research/data/generators/generate_adversarial_fcfs_workloads.py)

### 1. `natural_tenant_rotation_gap`

Interpretation:

- recurring large-prefix tenants/users return periodically
- between two requests from the same hot tenant, the system processes many other long-prefix tenant requests
- this mimics multi-tenant serving where the next useful family is not the most recently used one

### 2. `natural_periodic_refinement_gap`

Interpretation:

- iterative agent / analyst jobs revisit the same large context at later stages
- each revisit is separated by many side analyses or tool outputs
- this mimics staged workflows with delayed prefix reuse

These workloads are intentionally not grouped by family in the order that would favor `LRU`.

## Fixed Runtime Settings

- scheduler: `fcfs`
- model: `Qwen/Qwen2.5-7B-Instruct`
- max concurrency: `128`
- request rate: `inf`
- `mem_fraction_static=0.24`
- one GPU

## Sweep Knob

Sweep:

- `page_size in {16, 32, 64, 128}`

Why:

- small page size gives fine eviction granularity and often helps `LRU`
- larger page size makes each wrong eviction more expensive
- if the Belady-vs-LRU gap grows with page size, then the weak earlier result was partly a granularity effect

## Method

For each `(workload, page_size)` pair:

1. Run LRU under FCFS and collect `traces/lru.jsonl` and `benchmarks/lru.jsonl`
2. Compile a Belady plan from the LRU trace
3. Re-run the exact same workload under FCFS using Belady replay
4. Compare:
   - HBM hits / misses
   - HBM hit / miss rate
   - throughput
   - TTFT
   - ITL
   - end-to-end latency

## Recommended Runtime Budget

To keep total runtime reasonable on one H100:

- `2 workloads x 4 page sizes = 8 two-pass runs`
- use generated workload sizes directly
- skip remote analysis during serving runs and post-process later

This should stay in a manageable single-GPU budget while being much stronger than the earlier FCFS panel.
